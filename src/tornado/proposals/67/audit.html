Tornado Cash — Governance Proposal 67

A plain-language, source-cited analysis of the malicious proposal: what it does, exactly which contracts and roles it touches, and what it can and cannot reach — every fact verified on a mainnet fork.

Target Proposal #67 Network Ethereum mainnet Method on-chain reads + decompilation + Foundry fork tests Status being defeated (0 for / 623,201 against) Date 2026-06-29

Summary Verdict

HIGH — protocol treasury safe

Proposal 67 is a malicious proposal disguised as a fee/burn change. Its payload is a counterfeit Relayer Registry whose privileged functions answer to attacker-controlled vanity look-alike addresses instead of the DAO. If it passed, the attacker could disable the relayers that make private transactions possible. It cannot touch the treasury — we tried every asset-moving path on a fork and all of them fail.

Bottom line This is an attack on the protocol (a relayer kill-switch), not on the treasury. The ~7.32M TORN held by the DAO answers only to the real governance contract — an authority the spoofed addresses can never impersonate.

Glossary Plain terms used in this report

Governance proxy
The DAO's "brain" contract. A passed proposal runs as this contract, so it controls the treasury.
Relayer
An operator that processes private withdrawals. Must stake TORN to work. If unstaked, it stops.
Relayer Registry
The bookkeeping contract that records each relayer's stake and can zero it (nullifyBalance).
Vault
The contract that physically holds locked/treasury TORN. Releases funds only to the governance proxy.
Vanity look-alike
An address crafted to share the same first characters as a real one, so it looks identical at a glance.
delegatecall
Runs another contract's code inside your own storage/identity — how a proposal acts with the DAO's authority.
lockedBalance
Tornado voting power: TORN you locked in governance. Only you can unlock your own.
Quorum
Minimum total votes for a result to count — 100,000 TORN here.

Core trick The look-alike addresses

The payload's privileged accessors return addresses that share the first 15 hex characters with the real contracts, then diverge — designed to pass a quick visual check.

Mechanism What the proposal actually does

  1. The proposal is a contract (Tornado proposals always are). If executed, it swaps the DAO's Relayer Registry for the attacker's counterfeit one (0x0D0BE5…).
  2. In the counterfeit registry, the function nullifyBalance(relayer) — which zeroes a relayer's stake — is gated to the spoofed governance, i.e. the attacker, not the DAO.
  3. The attacker can then zero any relayer's staked TORN at will, removing the operators that enable private withdrawals. Reward flows are also redirected to the spoofed staking address.
  4. No function in the payload can move treasury assets (see below).

Note: the payload as currently deployed has no executeProposal() — it relies on a metamorphic redeploy (the May-2023 technique) to gain it before execution. As-is, execute(67) reverts (verified).

Treasury Why the funds are out of reach

The DAO treasury is ~7.32M TORN: 2.59M in the Vault (0x2F50…) and 4.73M held by the governance proxy itself. Both release only to the real governance proxy — an authority that is hardcoded, not based on any value the attacker can spoof.

Exhaustive attack — every vector failed Acting as the attacker on a fork, we attempted: Vault withdrawal, transferFrom on the Vault and the proxy, unlock of others' funds, and a proxy takeover via upgradeTo. All reverted.
Contrast — the real authority can drain everything Acting as the real governance proxy, the same fork test emptied the entire 7.32M TORN treasury (Vault + proxy). So the treasury is reachable — but only by a proposal that runs as the proxy and explicitly calls withdrawTorn + transfer. Proposal 67 contains no such calls.

Why the attacker avoided it: a proposal that obviously moves 7.32M TORN to a wallet is trivially spotted and voted down. The relayer-registry swap via look-alike addresses is the stealthy route — protocol sabotage that can survive a casual review.

Method How this was verified

  • On-chain reads — view calls against mainnet via a non-censoring RPC (publicnode); some public RPCs block Tornado contracts.
  • Decompilation — the unverified payload's bytecode, via Dedaub, published in pcaversaccio's gist.
  • Foundry fork tests — 16 tests executed against forked mainnet state, simulating the exact transactions an attacker (and the real proxy) would send. All pass:
    • Proposal67Content.t.sol — spoofed look-alike addresses (3)
    • Proposal67Powers.t.sol — attacker capability matrix (3)
    • Proposal67TreasuryDrain.t.sol — exhaustive treasury attack (6)
    • TornWriteActions.t.sol — Anticapture vote/delegate paths (4)
